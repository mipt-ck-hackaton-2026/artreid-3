# Справочник полей: `MIPT_hackathon_dataset.csv`

## Откуда взялся файл

Это выгрузка из **AmoCRM** (система, где компания ведёт заказы и клиентов). Каждая **строка** таблицы — **одна сделка** (в CRM её часто называют «лидом», но по смыслу это **заказ**, который связан с отгрузкой и доставкой).

Часть столбцов **скопирована из карточки сделки или карточки клиента** в момент подготовки датасета. Часть **посчитана отдельно** по **журналу изменений** в CRM: когда сотрудники переводили заказ на следующий этап, в системе сохраняются отметки времени — по ним восстановлены даты «передали в доставку», «выдали курьеру» и т.д.

Перед выдачей для учебы датасет **обезличен**: убраны персональные данные, а некоторые номера заменены на условные коды (`LEAD_…`, `CTR_…`, `MGR_…`), чтобы нельзя было восстановить конкретного человека, но строки по-прежнему можно считать и строить графики.

---

## Как читать «время» в этом файле

- **`…_ts` в конце имени** — как правило, **время в формате Unix timestamp**: целое число — это **сколько секунд прошло с полуночи 1 января 1970 года по всемирному координированному времени (UTC)** до соответствующего события. Такой формат удобен для программ: можно сравнивать числа, считать разницу в секундах. Чтобы перевести в обычную дату и время, нужен калькулятор timestamp или функция в Excel/Python (например, в Python: `datetime.utcfromtimestamp(значение)` с учётом часового пояса).

- **`sale_date`** — **календарная дата** (год-месяц-день) для того же момента, что и `sale_ts`: когда заказ **оформлен**, пошёл **в сборку и отправку** и **начал движение по воронке доставки** в учёте. Записана в удобном виде для фильтров и отчётов по дням; отдельного «второго события» за ней не стоит.

- Поля вида **`lead_Дата …`** — чаще всего **дата из соответствующего поля AmoCRM** (как её сохранили менеджеры или интеграции), в выгрузке они могут храниться числом; смысл указан в названии столбца.

## Описание столбцов

Ниже для каждого столбца: **что за информация**, **откуда она в датасете** (карточка сделки, карточка контакта, расчёт по журналу этапов или обезличивание).

| Поле | Описание |
|------|----------|
| `lead_id` | **Что это:** условный уникальный код строки (заказа) в этой выгрузке, после обезличивания (например `LEAD_0217`). **Откуда:** внутренний номер сделки в AmoCRM, преобразованный скриптом так, чтобы нельзя было сопоставить с живым номером в CRM. **Зачем:** связать строку с другими таблицами, не раскрывая реальный ID. Не стоит использовать как «числовой признак» в модели: величина кода сама по себе не несёт смысла. |
| `sale_ts` | **Что это:** момент, который в аналитике и у студентов удобно считать так: **человек оформил заказ**, заказ **пошёл собираться и отправляться**, и в **воронке доставки** он **начал двигаться** (вошёл в доставочный контур). Хранится как **Unix timestamp** (см. раздел «Как читать время»). **Откуда:** в датасете это время **восстанавливается из журнала смен этапов** в AmoCRM — фиксируется первый переход сделки на воронку «Доставка», что в нормальном процессе совпадает с описанным смыслом «оформили и пошли в доставку». **Смысл:** опорная отметка «старт заказа в доставке» для расчёта интервалов (`days_sale_to_handed` и др.) и календарных признаков. |
| `sale_date` | **Что это:** **дата календаря** (год-месяц-день) для того же смысла, что и `sale_ts`: оформление заказа, запуск сборки/отправки и **начало движения по воронке доставки**. **Откуда:** получена из `sale_ts` при формировании CSV. **Смысл:** удобно читать человеку и строить срезы по дням и месяцам без перевода timestamp. |
| `buyout_flag` | **Что это:** ответ «да/нет» на вопрос: **заказ в итоге дошёл до успешного получения и оплаты** по правилам, заложенным в выгрузку (`True`/`False` в файле может отображаться как 1/0 в зависимости от программы). **Откуда:** вывод по журналу этапов: если зафиксирован исход «товар получен» / «успешно реализовано», флаг истинный; если зафиксирован отказ при получении, возврат посылки или закрытие без реализации — ложный; иначе исход мог быть ещё не ясен. **Смысл:** типичная **целевая переменная**, если задача — предсказать выкуп. |
| `outcome_unknown` | **Что это:** признак **«финал заказа в данных ещё не определён»**: заказ не в успешном получении, не в явном отказе/возврате по правилам выгрузки, а **ещё в процессе** (в пути, ожидает и т.п.). **Откуда:** тот же расчёт жизненного цикла, что и для `buyout_flag`. **Смысл:** для обучения модели с жёстким «да/нет» по выкупу такие строки часто отделяют или исключают, потому что «правильного ответа» ещё нет. |
| `handed_to_delivery_ts` | **Что это:** момент времени (Unix timestamp), когда в журнале **впервые** появился этап **«Передан в доставку»** для этой сделки в воронке доставки. **Откуда:** из истории смены этапов в AmoCRM. **Смысл:** когда отправитель **передал заказ** в службу доставки по данным CRM. |
| `issued_or_pvz_ts` | **Что это:** самый **ранний** из двух моментов (если оба были): **когда посылку выдали курьеру** или **когда её приняли в пункт выдачи** — то есть когда заказ **покинул склад отправителя** в сторону клиента в логике CRM. **Откуда:** из журнала этапов; если случились оба события, берётся то, что произошло раньше. **Смысл:** когда посылка **ушла «в последнюю милю»** (к двери или в ПВЗ). |
| `received_ts` | **Что это:** момент времени (Unix timestamp), когда в журнале **впервые** зафиксирован **успешный исход получения** (клиент забрал и оплатил по правилам учёта). **Откуда:** история этапов AmoCRM. **Смысл:** когда заказ **закрылся успешно** с точки зрения доставки. |
| `rejected_ts` | **Что это:** момент времени, когда зафиксирован **отказ клиента при получении** (не забрал / отказался). **Откуда:** журнал этапов. **Смысл:** неуспешный исход на этапе выдачи. |
| `returned_ts` | **Что это:** момент времени, когда зафиксировано, что **посылка вернулась** (на склад / отправителю) по правилам воронки. **Откуда:** журнал этапов. **Смысл:** логистический возврат, не путать с отказом у двери без возвратной цепочки — здесь именно отметка «вернулась». |
| `days_sale_to_handed` | **Что это:** **сколько суток** прошло от `sale_ts` до `handed_to_delivery_ts`, с округлением до сотых. **Откуда:** вычислено при выгрузке из двух timestamp. **Смысл:** скорость передачи заказа в службу доставки после входа в доставочную воронку. Пустое значение, если не хватает одной из дат. |
| `days_handed_to_issued_pvz` | **Что это:** **сколько суток** от передачи в доставку до момента «у курьера / в ПВЗ» (`handed_to_delivery_ts` → `issued_or_pvz_ts`). **Откуда:** вычислено при выгрузке. **Смысл:** сколько длился этап от приёма службой доставки до выдачи в доставку клиенту. Отрицательные значения возможны, если в CRM этапы поставили **в нелогичном порядке по времени** (исправления задним числом и т.п.) — это артефакт данных, а не «отрицательная длительность» в реальности. |
| `days_to_outcome` | **Что это:** **сколько суток** от `sale_ts` до **закрытия** сделки по итогу в логике выгрузки (успех, отказ, возврат или «закрыто не реализовано» — в зависимости от того, что наступило). **Откуда:** вычислено из timestamp жизненного цикла. **Смысл:** общая длительность «от входа в доставку до финала» в данных. |
| `lifecycle_incomplete` | **Что это:** `True`/`False`: **не удалось** в журнале найти момент входа в доставочную воронку (`sale_ts` пустой), поэтому цепочка этапов **неполная** и производные «дней между этапами» могут быть пустыми. **Откуда:** флаг при сборке строки датасета. **Смысл:** маркер **качества строки**; такие заказы анализируют отдельно или исключают из метрик, где нужна полная шкала. |
| `current_status_id` | **Что это:** **числовой код этапа**, на котором сделка находилась **в момент снимка** при выгрузке из API AmoCRM. **Откуда:** стандартное поле сделки в CRM. **Смысл:** без справочника AmoCRM это «внутренний номер этапа»; по нему в отчётах обычно подставляют человеческие названия из `mappings.json` в репозитории проекта. |
| `closed_ts` | **Что это:** момент времени **закрытия** сделки по итогам в расчёте жизненного цикла (совпадает с успешным получением, отказом, возвратом или отдельным типом «закрыто не реализовано» — в зависимости от того, что произошло первым в правилах выгрузки). **Откуда:** вывод по журналу этапов. **Смысл:** когда в данных «закончилась история» заказа в доставке. |
| `lead_price` | **Что это:** сумма сделки **в рублях**, как она указана в AmoCRM в стандартном поле «бюджет/цена» сделки. **Откуда:** карточка сделки. **Смысл:** оценочная стоимость заказа в CRM (может отличаться от фактической оплаты, если поля не синхронизированы). |
| `lead_responsible_user_id` | **Что это:** условный код **менеджера по продажам**, ответственного за сделку (`MGR_…` после обезличивания). **Откуда:** стандартное поле сделки + псевдонимизация. **Смысл:** «кто вёл заказ» без раскрытия личности. |
| `lead_group_id` | **Что это:** числовой **идентификатор группы сделок** в AmoCRM (как в системе разделяют потоки сделок для отчётов). **Откуда:** стандартное поле сделки. **Смысл:** принадлежность к внутренней группе; расшифровка группы — в настройках CRM. |
| `lead_status_id` | **Что это:** код **текущего этапа воронки** сделки на момент выгрузки (может совпадать по смыслу с `current_status_id` в зависимости от версии API и маппинга). **Откуда:** карточка сделки. **Смысл:** этап воронки как число; название этапа — в справочнике CRM. |
| `lead_pipeline_id` | **Что это:** числовой код **воронки** (например, «Доставка»), в которой лежит сделка. **Откуда:** карточка сделки. **Смысл:** в учебном срезе часто почти одинаков для всех строк. |
| `lead_loss_reason_id` | **Что это:** если сделку закрыли **без успешной продажи**, в CRM может быть указана **причина** — это внутренний числовой код причины. **Откуда:** стандартное поле AmoCRM. **Смысл:** классификация проигрыша; может быть пустым, если сделка успешна или причина не заполнена. |
| `lead_created_at` | **Что это:** когда **создали карточку сделки** в AmoCRM (Unix timestamp). **Откуда:** стандартное поле сделки. **Смысл:** возраст сделки от создания; отличается от «входа в доставку» (`sale_ts`). |
| `lead_updated_at` | **Что это:** когда карточку сделки **последний раз обновляли** в CRM. **Откуда:** стандартное поле. **Смысл:** любое изменение полей увеличивает это время; само по себе не означает конкретный бизнес-этап. |
| `lead_closed_at` | **Что это:** когда сделку **закрыли** в CRM (успех или проигрыш) — время из карточки. **Откуда:** стандартное поле. **Смысл:** близко к «официальному закрытию» в CRM; может отличаться от расчётных `closed_ts`/`received_ts` из журнала этапов на секунды или в редких случаях расходиться, если данные правили вручную. |
| `lead_is_deleted` | **Что это:** признак, что карточку сделки **пометили удалённой** в CRM. **Откуда:** стандартное поле. **Смысл:** фильтрация мусорных или архивных записей. |
| `lead_Дата создания сделки` | **Что это:** **дата создания** сделки, как её хранят в **настраиваемом поле** с таким названием в AmoCRM (может дублировать или уточнять `lead_created_at`). **Откуда:** пользовательское поле сделки. **Смысл:** когда заказ завели с точки зрения бизнес-процесса; при расхождениях с `lead_created_at` разбирают оба источника. |
| `lead_Источник` | **Что это:** текстовое поле **источника заказа** (канал, партнёр, название потока — как заведено в CRM). **Откуда:** настраиваемое поле сделки. **Смысл:** откуда пришёл клиент с точки зрения менеджмента. |
| `lead_Состав заказа` | **Что это:** многострочный **текст с перечнем товаров**, артикулами, размерами и т.д., как заполнили в CRM. **Откуда:** настраиваемое поле сделки. **Смысл:** содержимое заказа для анализа номенклатуры (часто нужна обработка текста). |
| `lead_Квалификация лида` | **Что это:** оценка **качества лида** из **выпадающего списка** в карточке сделки. **Откуда:** настраиваемое поле AmoCRM. **Как заполняется:** **менеджер ОП (отдела продаж) после первого звонка** выставляет значение **вручную**; может **не заполнить** — тогда в данных будет пусто. **Важно:** делается **один раз** и в процессе **не меняется**. **Смысл:** субъективная, но стандартизированная метка «насколько заявка нам подходит» после первого контакта. |

| `lead_Модель телефона` | **Что это:** модель телефона, если её фиксируют в заявке. **Откуда:** настраиваемое поле сделки. **Смысл:** косвенный признак устройства клиента. |
| `lead_Проблема` | **Что это:** описание **запроса или симптома** (для медицинских/оздоровительных товаров — что беспокоит клиента), текст из CRM. **Откуда:** настраиваемое поле. **Смысл:** тематика заказа для анализа спроса. |
| `lead_Тариф Доставки` | **Что это:** название или код **тарифа доставки** (скорость, тип обслуживания). **Откуда:** настраиваемое поле. **Смысл:** условия перевозки. |

| `lead_Скидка` | **Что это:** размер или код **скидки** по заказу, как записано в CRM. **Откуда:** настраиваемое поле. **Смысл:** ценовая политика по строке. |
